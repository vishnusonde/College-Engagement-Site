How to Test Code Coupled to APIs or Databases

In the real-world, there's more to test than pure functions and React components. We have entire bodies of code that rely on databases, caches, APIs, and other infrastructural concerns. Depending on our testing strategy, our first task may be to apply some distance between two types of code: core core and infrastructure code. With the help of a layered architecture, we introduce testing options and expose ways in which we can develop a healthy testing strategy for code reliant on infrastructure like APIs or databases. In our humble introduction to Test-Driven Development, we learned the basics of TDD. You now know that you should start with a failing test, write the mininum amount of code to make it pass, and refactor it to make it cleaner. Red-green-refactor. Straightforward, right? Though it may take a while to internalize and start working this way, the physical aspect - the mechanics of it - are pretty easy to grasp. That's the good news. What's not good news is that when you get out there into the real world, you're going to encounter situations where you need to test code that relies on a variety of infrastructural dependencies: external APIs, services, databases, caches, webhooks, and so on.
